# VibeSec Review

> AI/vibe-coding으로 만든 웹앱을 배포 전후에 점검하는 **소유자 승인 기반 보안 리뷰 서비스** 기획안입니다.

VibeSec Review는 Lovable, v0, Cursor, Codex, Bolt 등으로 빠르게 만든 웹앱에서 자주 발생하는 보안 실수를 점검하고, 고객이 이해하기 쉬운 리포트와 수정 가이드를 제공하는 부업형 보안검사 웹사이트입니다.

초기 목표는 결제형 SaaS가 아니라 **문의형 랜딩페이지 + 신청폼 + 수동 결제 + Codex 기반 리포트 자동화 + 사람이 최종 검수**하는 MVP를 만드는 것입니다.

---

## 1. 핵심 포지셔닝

### 한 줄 설명

```text
AI로 만든 웹앱, 배포 전에 보안부터 확인하세요.
```

### 서비스 정의

VibeSec Review는 무단 해킹이나 공격 대행 서비스가 아닙니다.

- 의뢰자가 소유하거나 관리 권한을 가진 웹앱만 점검합니다.
- 점검 범위는 사전에 합의된 내용으로 제한합니다.
- 코드, 설정, 인증 흐름, 배포 상태, 데이터 접근 권한을 중심으로 리뷰합니다.
- 결과는 High / Medium / Low 위험도 리포트와 수정 가이드로 제공합니다.
- 수정이 필요한 경우 별도 옵션으로 Pull Request 또는 패치 제안을 제공합니다.

### 타깃 고객

| 고객 유형 | 주요 니즈 |
| --- | --- |
| 바이브코더 | 만든 웹앱이 안전한지 모름 |
| 1인 개발자 | Supabase/Firebase/Auth/API 설정이 불안함 |
| 노코드/AI 앱 제작자 | 배포는 했지만 보안 체크리스트가 없음 |
| 초기 스타트업 | 비싼 보안업체 전에 1차 점검이 필요함 |
| 프리랜서 개발자 | 납품 전 기본 보안점검 리포트가 필요함 |

## 3. MVP 전략

초반에는 카드결제나 복잡한 고객 대시보드 없이, 문의형 웹사이트로 시작합니다.

```text
랜딩페이지
→ 서비스 설명
→ 점검 범위 안내
→ 가격표
→ 신청폼
→ 수동 견적/상담
→ 계좌이체 또는 PayPal
→ Codex 기반 점검
→ 사람이 최종 검수
→ 리포트 전달
```

### MVP에서 하지 않을 것

- 카드결제/PG 연동
- 고객 대시보드
- 자동 취약점 스캐너 SaaS화
- 실시간 점검 자동화
- 무인 리포트 발송
- 무단 크롤링/무단 스캔
- 대량 요청 기반 테스트

## 4. 사업자 및 결제 운영 기준

초기에는 문의형 사이트와 무료/저가 베타로 수요를 확인합니다.

반복적으로 돈을 받고 서비스를 제공하기 시작하면 개인사업자 등록을 검토합니다.

### 추천 순서

```text
1. 웹사이트 제작
2. 무료/저가 베타 3~5건 진행
3. 리포트 템플릿 완성
4. 유료 반복 가능성 확인
5. 개인사업자 등록
6. 결제/세금계산서/통신판매업 여부 정리
```

### 초반 결제 방식

| 상황 | 추천 방식 |
| --- | --- |
| 국내 개인 고객 | 계좌이체 선불 |
| 해외 고객 | PayPal |
| 플랫폼 고객 | 크몽/프리랜서 플랫폼 에스크로 |
| 사업자 등록 후 | 세금계산서/현금영수증/PG 검토 |

### 결제 정책

```text
10만 원 이하: 100% 선결제
10만~50만 원: 50% 착수금 / 50% 리포트 전달 전 잔금
50만 원 이상: 계약서 + 착수금 + 중간금 + 잔금
```

초기 추천 상품은 **Basic Review 39,000원 런칭가 100% 선불**입니다.

## 6. 점검 범위

보안 기준은 독자 기준이 아니라 OWASP Top 10, OWASP API Security Top 10 등 일반적인 웹 보안 체크리스트를 참고하는 형태로 구성합니다.

| 영역 | 점검 내용 |
| --- | --- |
| 인증 | 로그인, 세션, 토큰 저장 위치, 로그아웃 |
| 권한 | 일반 유저가 관리자 기능에 접근 가능한지 |
| API | 민감 API 노출, 과도한 데이터 반환 |
| DB | Supabase RLS, Firebase rules, 공개 bucket |
| Secret | `.env`, API key, service role key 노출 |
| 배포 | CORS, 보안 헤더, HTTPS, admin route |
| 의존성 | npm/pnpm/yarn package 취약점 |
| 입력값 | XSS, injection 가능성, form validation |
| 파일 | 업로드 확장자, 공개 URL, storage rule |
| 로그 | 개인정보/토큰 로그 노출 여부 |

## 7. 금지 행위

웹사이트와 신청폼에 아래 내용을 명확히 표시합니다.

```text

다음 행위는 진행하지 않습니다.

- 무단 침투
- DDoS 또는 대량 요청

- 실제 결제 시도
- 악성코드 업로드
- 계정 탈취
- 권한 범위를 넘는 접근
- 제3자 서비스에 대한 무단 테스트
```

## 10. 신청폼 설계

신청폼은 단순 문의폼이 아니라, 권한 확인과 점검 범위 합의를 받는 절차입니다.

| 필드 | 예시 |
| --- | --- |
| 이름/닉네임 | 홍길동 |
| 연락처 | 이메일, 카카오톡, 디스코드 |
| 서비스 이름 | My AI App |
| 서비스 URL | `https://example.com` |
| GitHub repo | 공개/비공개 여부 |
| 사용 기술 | Next.js, Supabase, Firebase 등 |
| 점검 희망 상품 | Basic / Standard / Fix PR |
| 운영 상태 | 개발중 / 스테이징 / 운영중 |
| 테스트 계정 제공 가능 여부 | 가능/불가 |
| 개인정보 처리 여부 | 있음/없음/모름 |
| 결제 방식 | 계좌이체 / PayPal / 플랫폼 |
| 권한 확인 | 본인은 점검 대상의 소유자 또는 관리 권한자입니다. |
| 범위 동의 | 합의된 범위 밖 테스트를 요청하지 않습니다. |

## 11. Codex 활용 워크플로우

### 내부 운영 흐름

```text

→ 체크리스트별 결과 정리
```

| 재점검 | 수정 후 체크리스트 확인 | 최종 납품 |

## 12. Codex 프롬프트 템플릿

### Basic Review 점검용

```text
너는 승인 기반 웹앱 보안 리뷰어다.
이 repo는 의뢰자가 소유권 또는 관리 권한을 가진 프로젝트다.

목표:
- 침투 테스트가 아니라 코드/설정/배포 보안 리뷰를 수행한다.
- 실제 공격, 대량 요청, 데이터 변조는 하지 않는다.
- 발견 항목은 High / Medium / Low로 분류한다.
- 각 항목에 영향, 근거, 수정 가이드, 우선순위를 제공한다.

중점 점검:
1. 인증/세션
2. 접근제어
3. API route
4. 환경변수/secret 노출
5. Supabase/Firebase 설정
6. 파일 업로드
7. CORS/security headers
8. dependency 취약점
9. 개인정보 노출 가능성
10. 배포 설정

출력:
- executive summary
- finding list
- remediation checklist
- client-friendly explanation
```

### 리포트 생성용

```text
다음 점검 결과를 고객 전달용 보안 리포트로 정리해라.

조건:
- 과장하지 말 것
- 실제 확인된 것과 가능성을 구분할 것
- 공격 악용 절차를 자세히 쓰지 말 것
- 개발자가 바로 수정할 수 있는 가이드 중심으로 작성할 것
- High / Medium / Low 기준으로 정렬할 것
```

## 13. 리포트 포맷

### 전체 구조

```text
1. 점검 개요
2. 점검 대상
3. 점검 범위
4. 제외 범위
5. 요약 결과

7. 위험도별 우선순위
8. 수정 가이드

10. 면책 및 제한사항
```

### 발견 항목 템플릿

```text
[High] Supabase RLS 미설정으로 데이터 노출 가능성

대상:
- users 테이블
- public API query

설명:
현재 클라이언트에서 특정 조건 없이 데이터를 조회할 수 있는 구조입니다.

영향:
일반 사용자가 다른 사용자의 데이터를 조회할 가능성이 있습니다.

재현 조건:
소유자 승인 범위 내 테스트 계정 기준으로 확인

수정 권장:
- RLS 활성화
- 사용자별 owner_id 검증 policy 추가
- service role key 클라이언트 노출 금지

우선순위:
즉시 수정 권장
```

```text
→ 이메일/DM으로 전달
```

접수 API는 브라우저에 메일 서비스 키를 노출하지 않는다. 운영 배포에서는 `RESEND_API_KEY`, `INTAKE_FROM_EMAIL`, `INTAKE_TO_EMAIL` 환경변수를 서버에 설정한다.

## 17. FAQ 초안

### Q. 제 사이트를 실제로 해킹하나요?

아니요. 본 서비스는 무단 침투나 공격 대행이 아닙니다. 소유자 승인 기반으로 코드, 설정, 인증 흐름, 배포 상태를 점검하고 필요한 경우 테스트 계정과 합의된 범위 안에서만 제한적인 수동 검증을 수행합니다.

### Q. GitHub repo를 꼭 줘야 하나요?

Basic은 공개 URL만으로도 일부 점검이 가능합니다. 다만 코드와 설정까지 확인하려면 GitHub repo 접근 권한이 필요합니다.

### Q. 취약점이 발견되면 고쳐주나요?

기본 상품은 리포트 제공까지 포함됩니다. 수정 PR은 별도 옵션으로 진행됩니다.

### Q. AI가 검사하면 믿을 수 있나요?

AI는 코드와 설정을 빠르게 분석하는 보조 도구로 사용됩니다. 최종 리포트는 사람이 검수한 뒤 전달됩니다.

## 18. 약관에 포함할 핵심 내용

```text
- 의뢰자는 점검 대상의 소유자 또는 관리 권한자여야 한다.
- 점검 범위는 신청서와 합의 문서에 명시된 범위로 제한된다.
- 무단 침투, 대량 요청, 데이터 변조, 개인정보 다운로드는 제외된다.
- 광범위한 침투테스트는 기본 상품이 아니며, 권한·세션·파일 접근의 제한적 재현 확인만 별도 합의한다.
- 점검 결과는 특정 시점 기준이며 완전한 보안을 보장하지 않는다.
- 의뢰자가 제공한 접근정보는 점검 목적 외 사용하지 않는다.
- 리포트는 의뢰자 내부 개선 목적으로 제공된다.
- 제3자 시스템에 대한 무단 점검 요청은 거절된다.
```
