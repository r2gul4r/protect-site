# VibeSec Review MVP Plan

## 목표

README 기준 MVP는 `랜딩페이지 + 신청폼 + 샘플 리포트 + 수동 결제 안내 + Codex 기반 점검 초안 + 사람 최종 검수`다.

## 0단계: MVP 제품 결정

- 로그인/회원가입은 MVP에 넣지 않는다.
- 고객 대시보드, 계정 관리, 리포트 보관함은 유료 반복 수요가 확인된 뒤 검토한다.
- 초기 의뢰는 `/apply` 신청폼으로 받고, 제출 내용은 운영자 이메일로 전달한다.
- 초기에는 DB를 두지 않는다. 필요하면 이메일을 보고 Google Sheet 또는 Notion에 수동 기록한다.
- Google OAuth, Google Sheets API 직접 연동은 MVP에서 제외한다.
- 반복 접수가 생기면 2차로 Make/Zapier 등을 써서 이메일을 Google Sheet로 자동 기록한다.
- 직접 Google API 연동은 신청량, 개인정보 보관 정책, 접근권한 설명이 정리된 뒤 3차로 검토한다.
- 결제는 계좌이체, PayPal, 플랫폼 에스크로 안내까지만 다루고 카드결제/PG는 제외한다.
- 신청폼은 권한 확인과 범위 동의 체크가 없으면 제출할 수 없게 설계한다.
- 첫 구현 목표는 방문자가 서비스 이해, 가격 확인, 점검 범위 확인, 신청폼 도달까지 끝내는 정적 MVP다.
- 헤더 사운드 토글처럼 보안 리뷰 서비스와 무관한 장식 기능은 넣지 않는다.
- 샘플 리포트는 임의 파일 다운로드가 아니라 페이지 안에서 전달물 구조와 finding 예시를 보여준다.
- 신청폼은 같은 화면에서 `/api/inquiries`로 제출하고, 메일 발송 키는 서버 환경변수로만 관리한다.

## 1단계: 제품 계약 고정

- 서비스명은 `VibeSec Review`로 시작한다.
- 한 줄 카피는 `AI로 만든 웹앱, 배포 전에 보안부터 확인하세요.`를 사용한다.
- 금지 행위와 소유자 승인 조건을 모든 핵심 페이지와 신청폼에 반복 노출한다.
- MVP 밖 기능을 명확히 제외한다: 카드결제, 대시보드, 자동 스캐너 SaaS, 무인 리포트 발송.

## 2단계: 사이트 골격

- `/`: 서비스 소개, CTA, 상품 요약, 승인 기반 점검 원칙.
- `/pricing`: Basic, Standard, Fix PR Add-on 가격과 포함 범위.
- `/scope`: 점검 범위, 제외 범위, 금지 행위.
- `/apply`: 신청폼과 권한/범위 동의 체크.
- `/sample-report`: 고객 공개용 샘플 finding과 리포트 구조.
- `/faq`: 해킹 여부, repo 제공, AI 검수 신뢰성, 수정 PR 안내.
- `/terms`, `/privacy`: 운영 전 초안 수준으로 작성하되 법률 검토 필요 문구 포함.

## 메인페이지 시안 방향

1. 신뢰 우선형: 밝은 배경, 실제 리포트 미리보기, 명확한 CTA 중심.
2. 개발자 도구형: 코드/설정 체크리스트와 위험도 배지가 보이는 실무형 UI.
3. 출시 전 점검형: 배포 직전 체크포인트와 Basic Review 상품을 첫 화면에 강조.
4. 스코프 안전형: 소유자 승인, 금지 행위, 비침투형 리뷰 원칙을 전면에 배치.
5. 프리미엄 리포트형: 사람 검수 리포트와 Fix PR 옵션을 고급 컨설팅 느낌으로 표현.

## 3단계: 신청/접수 흐름

- 신청폼 필드는 이름, 연락처, 서비스 URL, repo, 기술스택, 상품, 운영 상태, 테스트 계정 가능 여부, 개인정보 처리 여부, 결제 방식, 권한 확인, 범위 동의를 포함한다.
- 초기 저장은 서버리스 접수 API의 이메일 전송으로 시작하고, 필요하면 수신 메일을 Google Sheet/Notion에 수동 기록한다.
- 접수 후 고객 안내 메시지는 점검 가능 여부 확인, 결제 안내, 범위 확정 순서로 보낸다.
- Standard는 테스트 계정과 합의된 범위 안에서 권한·세션·파일 접근의 제한적 수동 검증을 포함할 수 있다.
- 무단 스캔, 부하 테스트, 파괴적 테스트, 개인정보 다운로드는 항상 제외한다.

## 4단계: 리포트 운영 템플릿

- Basic/Standard 점검 프롬프트를 분리한다.
- Supabase, Firebase, Auth, API route, 파일 업로드, CORS/security header, dependency, secret 노출 체크리스트를 만든다.
- 리포트는 High / Medium / Low와 수정 우선순위 중심으로 생성한다.
- 사람이 오탐 제거, 표현 수위 조절, 최종 납품 판단을 맡는다.

## 5단계: 샘플 검증

- 테스트용 앱 하나를 대상으로 실제 Basic Review 흐름을 리허설한다.
- 샘플 리포트 1개를 작성하고 공개 가능한 수준으로 민감 정보를 제거한다.
- 신청폼 제출부터 리포트 전달까지 수동 운영 시간이 얼마나 드는지 측정한다.
- 반복 유료 수요가 보이면 사업자등록, 결제/세금 처리, 개인정보 처리위탁 검토로 넘어간다.

## 우선순위

1. `AGENTS.md`와 MVP 범위 고정
2. Next.js 프로젝트 생성
3. 핵심 페이지 8개 작성
4. 신청폼과 접수 알림 연결
5. 샘플 리포트와 점검 프롬프트 작성
6. 테스트 앱으로 전체 흐름 리허설

## 검증 기준

- MVP 밖 기능이 구현 계획에 섞이지 않았는가
- 무단 침투나 공격 대행으로 오해될 문구가 없는가
- 신청폼에서 소유자 승인과 범위 동의를 받는가
- 리포트가 고객 친화적이면서도 과장 없이 수정 가능한 정보를 주는가
- 외부 연동이나 개인정보 저장이 추가될 때 보안 요구사항이 먼저 반영되는가
